## 本当に危険な CONVERT\_IMPLICIT 再現手順

> ※ datetime では再現しづらかった「暗黙の型変換による Scan」を
>
> **int + varchar** で _確実に_ 再現します。
>
>

---


## なぜ int を使うのか（前提）


SQL Server の **データ型優先順位**では、


```plain text
int > varchar / nvarchar
```


となっています。


この場合、比較式


```sql
varchar 列 = int パラメータ
```


は **列側が int に変換される**ため、


✅ **列側 CONVERT\_IMPLICIT 発生**


✅ **SARGable ではなくなる**


✅ **Index Seek 不可 → Index Scan**


という「本物の地雷」になります。


---


## 再現環境（完全手順）


### 1️⃣ テーブル作成（varchar 列にインデックス）


```sql
DROP TABLE IF EXISTS dbo.Users;
GO

CREATE TABLE dbo.Users
(
    UserId varchar(20) NOT NULL,
    Dummy  char(100) NOT NULL DEFAULT 'x'
);
GO
```


### 2️⃣ 大量データ投入（差が出るよう多めに）


20万件、Dockerでローカルにデバイスある状態で、２０分ほど・・・


```sql
DECLARE @i int = 1;

WHILE @i <= 200000
BEGIN
    INSERT INTO dbo.Users (UserId)
    VALUES (CAST(@i AS varchar(20)));

    SET @i += 1;
END;
GO
```


### 3️⃣ インデックス作成


2,3秒・・・


```sql
CREATE INDEX IX_Users_UserId ON dbo.Users(UserId);
GO
```

## 一見「問題なさそうな SQL」（地雷）


```sql
SELECT *
FROM dbo.Users

```

- ✅ SQL エラーなし
- ✅ 実行結果は正しい
- ❌ **Index があるのに遅い**

## 実行プランで見える決定的証拠


### ❌ 実行プラン

- **Index Scan**
- Predicate に以下が表示される

```plain text
CONVERT_IMPLICIT(int,[dbo].[Users].[UserId],0) = @id
```


![image.png](/images/image.png)


![image.png](/images/image.png)


### ここがポイント


✅ **列側（UserId）に CONVERT\_IMPLICIT**


❌ B-tree を使えない


❌ 全行評価 → **Index Scan**


👉 **これが本当に危険な CONVERT\_IMPLICIT**


---


## なぜこうなるのか（整理）


| 項目       | 型       |
| -------- | ------- |
| UserId 列 | varchar |
| @id      | int     |


➡ SQL Server は **int を優先**


➡ **varchar 列を int に変換**


---


## 正しい修正方法（明示変換）


### ✅ 条件を char に揃える


```sql
SELECT *
FROM dbo.Users
WHERE UserId = '150000';
```


---


## 修正後の実行プラン

- ✅ **Index Seek**
- ✅ Seek Predicate：

```plain text
[Users].[UserId] = @id
```


✅ CONVERT\_IMPLICIT 消滅


✅ IO 激減


✅ 実行時間が安定

## IO の差を数値で確認（任意）


```sql
SET STATISTICS IO ON;
GO

-- 悪い例
SELECT *
FROM dbo.Users
WHERE UserId = 150000;



-- 良い例
SELECT *
FROM dbo.Users
WHERE UserId = '150000';

GO
```


👉 **logical reads が桁違いになる**のを確認できます。


---


補足）


※ 暗黙の型変換が本当に問題になるのは、varchar 列と int パラメータの比較など、「列側が高優先度型に変換される場合」です。
datetime と文字列の比較では、SQL Server が定数側を変換するため、同じ問題は再現しません。


---


## まとめ（ここまでの結論）

- ✅ **CONVERT_IMPLICIT は常に悪ではない**
- ❌ **列側に出た瞬間アウト**
- ✅ datetime は安全寄り
- ❌ **varchar 列 × int パラメータは実務最頻出地雷**
- ✅ 実行プランを見ないと絶対に分からない
